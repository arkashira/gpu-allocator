## 1. Acceptance Criteria  
(Each bullet is measurable and testable)

1. **UI Rendering** – The policy configuration page loads in ≤ 2 s and displays a form with the following fields:  
   *Policy Name* (text, required)  
   *GPU Limit per Tenant* (integer, ≥ 1, required)  
   *Max Concurrent Jobs* (integer, ≥ 0, required)  
   *Time‑Window (optional)* (HH:MM‑HH:MM, optional)  
   *Enable/Disable* (toggle, default enabled)  

2. **Create / Update Policy** –  
   * Submitting a valid form for a new policy returns HTTP 201 and the policy appears in the list with the exact values.  
   * Submitting a valid form for an existing policy returns HTTP 200 and updates the list entry accordingly.  

3. **Real‑time Enforcement** – When a policy is active, any scheduling request that would exceed the defined limits is rejected with HTTP 429 and a JSON body containing `{"error":"policy_violation","detail":"GPU limit exceeded"}`.  

4. **Alerting** – On a policy violation, an alert message is sent to the configured channel (e.g., Slack webhook) within ≤ 5 s of the violation.  

5. **Persistence** – After a service restart, the UI still lists all previously created policies and enforcement continues unchanged.  

6. **Authorization** – Only users with the `admin` role can access the policy UI; other roles receive HTTP 403 when attempting to GET or POST the policy endpoint.  

7. **Input Validation** –  
   * Client‑side: negative numbers, non‑integer values, or missing required fields trigger inline error messages before the form can be submitted.  
   * Server‑side: the same invalid inputs return HTTP 400 with a JSON body `{"errors":[...]"}` describing each field error.
