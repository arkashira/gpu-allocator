## 1. Acceptance Criteria
1. **UI Rendering** – The policy configuration page loads within 2 s and displays a form with fields: *Policy Name*, *GPU Limit per Tenant*, *Max Concurrent Jobs*, *Time‑Window (optional)*, and *Enable/Disable* toggle.
2. **Create / Update Policy** – Submitting a valid form creates a new policy (HTTP 201) or updates an existing one (HTTP 200) and the policy appears in the policy list with the correct values.
3. **Real‑time Enforcement** – When a policy is active, any scheduling request that would exceed the defined limits is rejected with HTTP 429 and a descriptive error message.
4. **Alerting** – On policy violation, an alert is emitted to the configured notification channel (e.g., email, Slack webhook) within 5 s of the violation.
5. **Persistence** – Policies survive a service restart; after restart, the UI shows the same policies and enforcement continues unchanged.
6. **Authorization** – Only users with the *admin* role can access the policy UI; other roles receive HTTP 403 when attempting to load the page or submit changes.
7. **Input Validation** – Invalid inputs (negative numbers, non‑integer limits, missing required fields) are rejected client‑side with inline error messages and server‑side with HTTP 400.
