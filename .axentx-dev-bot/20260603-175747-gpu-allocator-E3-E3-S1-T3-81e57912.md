## 1. Acceptance Criteria
1. **Pattern Detection** – The detector flags a GPU allocation session as suspicious when **any** of the following conditions is met within a configurable sliding window (default 5 min):
   - average GPU utilization > `config.util_threshold` (%)
   - allocation duration > `config.duration_threshold` (seconds)
   - client IP resolves to a country ≠ `config.allowed_country` for the datacenter.
2. **Alert Persistence** – On detection, an `Alert` record is created in the alerts store with fields: `id`, `allocation_id`, `severity='high'`, `reason` (one of *high_util*, *long_duration*, *geo_mismatch*), `timestamp`.
3. **Real‑time Delivery** – The alert is published to the `notifications/diversion` channel **≤ 2 seconds** after the `Alert` record is persisted.
4. **Idempotency** – Only one alert may exist per `allocation_id`. Subsequent detections for the same allocation are ignored unless the existing alert is resolved/expired.
5. **Configurable Thresholds** – All thresholds (`util_threshold`, `duration_threshold`, `allowed_country`, `window_seconds`) are read from `config.yaml` and can be overridden at runtime via the `DiversionDetector.update_config()` API.
6. **Observability** – Prometheus metrics are emitted:
   - `diversion_detected_total{reason,datacenter}`
   - `diversion_alert_sent_total{reason,datacenter}`
   - `diversion_false_positive_total{reason,datacenter}`
7. **Security** – Alert payloads are signed with the system HMAC key; any consumer must be able to verify the signature using the shared secret.
